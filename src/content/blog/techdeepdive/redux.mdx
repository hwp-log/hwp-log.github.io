이전글에서 React가 "복잡한 UI를 효율적으로 그리는 문제"를 해결하기 위해 탄생한 과정을 살펴보았습니다. 
그렇다면 React가 UI 렌더링을 해결하는 동안, "상태를 어디에 두고, 누가 바꾸고, 어떻게 추적할지" 고민은 없었을까요?

이번 글에서는 

•&nbsp; 상태관리의 문제가 무엇이었는지,  
•&nbsp; 그 문제를 해결하기 위해 Flux 패턴이 어떻게 등장했는지,  
•&nbsp; 그리고 Flux에서 Redux로 어떻게 발전했는지를 알아 보도록 하겠습니다.  

#### 1. MVC의 패턴의 한계

React가 등장하기 이전부터, Facebook은 MVC 스타일 구조와 데이터 흐름 패턴을 많이 사용하고 있었습니다.  
이 방식은 초기에 잘 동작했습니다.

```jsx
① View -> ② Controller -> ③ Model -> ④ View
                            
① 사용자가 화면에서 입력
② Controller가 입력을 받아 처리
③ Model의 데이터를 갱신
④ 갱신된 데이터로 View를 다시 출력
```

그러나 기능이 많아질수록 점점 한계를 드러내기 시작했습니다.  
그 한계는 기능간의 영향을 주고 받는 <u style="color: red; font-weight: bold;">연쇄적인 업데이트(Cascading Updates)</u> 많아진 점 이였습니다.

```html
기능1 : ① View -> ② Controller -> ③ Model -> ④View -> Controller...
                                                ↓  ⑤ (기능②의 Controller에도 영향)
기능2 :                                   ⑥ Controller -> ⑦ Model -> ⑧ View -> Controller...
                                                                         ↓ ⑨ (기능③의 Controller에도 영향)
기능3 :                                                                  반복...

① 기능1의 View에서 사용자가 입력
② 기능1의 Controller가 입력을 받아 처리
③ 기능1의 Model 데이터가 갱신됨
④ 갱신된 데이터로 기능1의 View가 업데이트됨

⑤ 기능1의 View 변경이 기능2의 Controller에도 영향을 줌 [Cascading Updates]
⑥ 기능2의 Controller가 의도하지 않은 갱신을 받아 처리함
⑦ 기능2의 Model 데이터가 갱신됨
⑧ 기능2의 View가 잘못된 상태로 출력됨

⑨ 기능2의 View 변경이 기능3의 Controller에도 영향을 줌 [Cascading Updates]

반복...

```

핵심은 이런 <u style="color: red; font-weight: bold;">연쇄적인 업데이트(Cascading Updates)</u>가 여러곳에서 동시에 일어난다는 점입니다.

물론, 기능이 적은 상태에서는 많은 데이터를 관리하지 않기 때문에 추적이 가능했을 것입니다.  
하지만 대규모 앱이 되면서 이러한 영향이 많아지게 되자, <u style="color: red; font-weight: bold;">개발 및 유지보수에서 큰 문제</u>로 떠오르게 됩니다.

#### 2. '안 읽은 메시지 버그' — 상태 관리 실패의 대표 사례

이러한 문제가 실제로 터진 대표적인 사례가 있었습니다. 바로 페이스북에서 "실제로 새 메시지가 없는데도, 새 메시지가 있다고 사용자에게 알리는 채팅 버그" 입니다. 
이는 앞서 말한 "연쇄적인 업데이트 내용"이 서로 영향을 주면서, '컨트롤러'와 '모델'에 값이 갱신되고, 어느 부분에서 값이 바뀌었는지 알 수 없는 문제였습니다.

![Unread_mesages](../../../assets/20260320/Unread_mesages.png)

즉, 메시지를 담당하는 모델은 '읽었다'고 처리해도, 연쇄적으로 영향을 주는 다른 모델들이 아직 '안 읽었다'는 값을 가지고 있을 수 있는 구조였습니다.  

그래서 '안읽은 모델이 -> 읽은 모델' 로 영향을 줘 버리면 예측할 수 없는 상태 불일치를 만들어냈고, 이것이 "읽었는데 안 읽었다고 뜨는" 버그로 나타난 것입니다.

이렇듯 문제의 원인은 코드의 특정 위치가 아니라 <u style="color: red; font-weight: bold;">"모델들이 다른기능의 모델에 영향을 주는 구조"</u> 였습니다.   
이 문제를 계기로 Facebook 팀은 "이렇게 구조를 가져서는 안되겠다"고 판단하게 됩니다.

#### 3. Flux 패턴 — "예측 가능한 단방향으로 만들자"

그리고 이를 해결하고자 등장한 패턴이 Flux 패턴입니다.
 
![flux](../../../assets/20260320/flux.png)
 
정리하자면 아래와 같습니다.

[1] 기존 MVC에서는 연쇄적인 업데이트가 동시에 일어나, 어느 Model에서 값이 변경되었는지 알 수 없었습니다.  
[2] Flux는 이를 Dispatcher와 Store로 해결합니다.  
[3] Dispatcher는 액션을 Store에 전달하는 단일통로이며, <u>한 번에 하나의 Action</u>만 처리하도록 강제합니다.  
[4] <u style="color: red; font-weight: bold;">이전 Action이 완전히 끝나야 다음 Action이 처리됩니다.</u>  
[5] <u style="color: orange; font-weight: bold;">Store는 상태를 한 곳에서 관리하여, "안 읽은 메시지", "채팅 목록" 등의 같은 기능들이 Store에서 값을 읽게 됩니다.</u>

결론적으로는  

[1] Dispatcher가 처리 순서를 지정 -> <u style="color: red; font-weight: bold;">(동시에 영향을 주던 문제 해결)</u>  
[2] Store가 상태를 한 곳에서 관리 -> <u style="color: orange; font-weight: bold;">(어디서 바뀌었는지 모르는 문제 해결)</u>

이렇게 제어가 안되던 부분을 flux 패턴으로 개선하게 된 것입니다.

이 모든 과정은 항상 Action -> Dispatcher -> Store -> View 라는 단방향 흐름 안에서 일어나기 때문에,  
기존의 <u style="color: red; font-weight: bold;">예상치 못한 곳에서 값이 튀는 현상</u>을 크게 줄이고 <u style="color: orange; font-weight: bold;">Store에서 상태값을 제어</u>하여, 유지/보수의 효율을 올려줍니다.

#### 4. Flux에서 Redux로 — "더 단순하게"
 
Flux가 "상태 흐름을 단방향으로 고정하자"는 핵심 아이디어를 제시했다면, Redux는 그 아이디어를 더 단순하고 예측 가능하게 구현한 라이브러리입니다. Redux는 Flux의 개념을 계승하면서도 몇 가지 중요한 차이를 두었습니다.

그 차이에 대해 설명 해보겠습니다.
 
[1] **여러 Store 통합** 

Flux에서는 여러 개의 Store가 존재할 수 있었습니다. 채팅 Store, 알림 Store, 유저 Store 등등... 도메인별로 있었죠.  
<u style="font-weight: bold;">Redux는 이것을 하나의 Store로 통합했습니다.</u> 앱의 모든 상태가 하나의 객체 트리에 담기게 되면서, 상태를 찾으러 여러 Store를 돌아다닐 필요가 없어진 것입니다.
 
[2] **Dispatcher 흡수** 

Flux에서는 Dispatcher가 별도의 존재로 Action을 Store에 전달하는 역할을 했습니다.
<u style="font-weight: bold;">Redux는 이 역할을 Store가 흡수</u>하여, Store 자체가 dispatch 메서드를 가지게 했습니다.
별도의 중간 다리 역할이 사라지면서, 구조가 더욱 단순해진 것입니다.


[3] **Reducer의 도입**  

Redux에서 상태를 실제로 바꾸는 것은 Reducer라는 함수입니다.  
<u style="font-weight: bold;">"현재 상태" 와 "Action"을 받아서, "새로운 상태"를 리턴</u>합니다.  
<u style="color: red; font-weight: bold;">이때 기존 상태를 직접 바꾸지 않고, 항상 새로운 객체를 만들어서 반환</u>한다는 것이 핵심입니다.

왜 이렇게 설계했을까요?  
"현재 상태를 직접 바꾸지 않는다"는 규칙(불변성, immutability)이 있으면, 상태가 언제 어떻게 바뀌었는지 추적할 수 있기 때문입니다. 이전 상태와 새 상태가 별도의 객체로 존재하니까, 비교도 쉽고 되돌리기도 가능해집니다.

그럼 코드로 하나씩 짚어보겠습니다.

#### 5. Flux와 Redux, 코드로 비교하기

*아래의 번호는 실행순서입니다.
 
```jsx
// Flux: Dispatcher가 따로 있고, Store가 여러 개
import { Dispatcher } from 'flux';
 
// [2] Dispatcher — 별도의 존재로 Action을 Store에 전달
const AppDispatcher = new Dispatcher();
 
// [3] Store가 여러 개 — 채팅과 알림이 각각 따로

// [3-1]채팅 Store
let messages = [];
const ChatStore = {
  dispatchToken: AppDispatcher.register((action) => {
    if (action.type === 'READ_MESSAGE') {
      messages = messages.map(msg =>
        msg.id === action.messageId ? { ...msg, read: true } : msg
      );
    }
  })
};

// [3-2]알림 Store
let unreadCount = 0;
const NotificationStore = {
  dispatchToken: AppDispatcher.register((action) => {
    // 채팅-Store가 먼저 처리되길 기다림
    AppDispatcher.waitFor([ChatStore.dispatchToken]);
    // 채팅-Store의 변수를 직접 참조
    unreadCount = messages.filter(msg => !msg.read).length;
  })
};
 
// [1]Action 발행 — Dispatcher를 거쳐야 함
AppDispatcher.dispatch({ type: 'READ_MESSAGE', messageId: 1 });
```
 
```jsx
// Redux: Store 하나, Dispatcher 통합 됨, Reducer가 처리
import { createStore, combineReducers } from 'redux';

// [2] Reducer — 상태를 바꾸는 순수 함수
//     규칙 1: 파라미터로 받은 state만 봄 (외부 변수 참조 금지)
//     규칙 2: 새로운 객체를 만들어서 리턴 (기존 state 수정 금지)
//     규칙 3: 같은 입력이면 항상 같은 출력 (순수 함수)

function chatReducer(state = { messages: [] }, action) {
                    // ↑ 이게 이 Reducer의 state
  switch (action.type) {
    case 'READ_MESSAGE':
      return {
        messages: state.messages.map(msg =>
          msg.id === action.messageId ? { ...msg, read: true } : msg
        )
      };
    default:
      return state;
  }
}

function notificationReducer(state = { unreadCount: 0 }, action) {
                            // ↑ 이게 이 Reducer의 state
  switch (action.type) {
    case 'READ_MESSAGE':
      return { unreadCount: state.unreadCount - 1 };
      // ↑ Flux때와 달리 chatReducer의 존재를 모름. 자기 state만 처리.
    default:
      return state;
  }
}

// [3] Store는 하나 — Reducer를 합쳐서 하나로 관리
const store = createStore(combineReducers({
  chat: chatReducer,
  notification: notificationReducer
}));

// [1] Action — Store에 직접 dispatch
store.dispatch({ type: 'READ_MESSAGE', messageId: 1 });
```

차이점은 아래와 같습니다.
```
Flux:   [1] Action -> [2] Dispatcher -> [3] Store
Redux:  [1] Action -> [2] Reducer    -> [3] Store
```
 
이 두 코드의 차이는 디버깅에서 확실히 드러납니다.
 
```
Flux에서 버그 나면:
  -> Action 확인
  -> Dispatcher 확인
  -> ChatStore 확인
  -> NotificationStore가 ChatStore를 참조하고 있네? (*Flux에서는 타 Store 참조여부를 확인)
  -> waitFor 순서 확인
  -> Store 여러 개를 줄줄이 따라가야 함
 
Redux에서 버그 나면:
  -> Action 확인
  -> 해당 Reducer 하나만 확인
  -> 끝.
```
 
즉 Flux에서는 Store 간 참조가 가능했기 때문에, <u style="color: red; font-weight: bold;">오류가 1번 Store에서 터져도 원인은 3번 Store에 있을 수 있습니다.</u>   
Redux의 Reducer는 구조적으로 다른 Reducer의 state에 접근할 수 없기 때문에, <u style="color: red; font-weight: bold;">오류가 난 곳이 곧 원인</u>이 있는 곳입니다.
 
#### 6. Redux의 세 가지 원칙
 
Redux의 설계를 관통하는 원칙은 세 가지입니다.
 
**1. 상태는 하나의 Store에만 존재합니다.**  
여러 곳에 상태가 흩어져 있으면 "안 읽은 메시지 버그"가 재현됩니다. 하나의 Store에 모아두면 "지금 앱의 상태가 뭐지?"라는 질문에 한 곳만 보면 됩니다.
 
**2. 상태는 Action을 통해서만 바꿀 수 있습니다.**  
아무 컴포넌트에서나 직접 상태를 바꿀 수 없습니다. 반드시 Action이라는 객체를 발행해야 합니다. 이 규칙 덕분에 "누가 이 상태를 바꿨지?"가 항상 추적 가능합니다.
 
```jsx
// Action은 평범한 JavaScript 객체입니다
{ type: 'PLUS' }
{ type: 'ADD_TODO', text: 'Redux 공부하기' }
{ type: 'SET_USER', payload: { name: 'Engineering-Log' } }
```
 
**3. 상태 변경은 순수 함수(Reducer)로만 합니다.**  
같은 입력이면 항상 같은 출력. 외부에 영향을 주지 않습니다. 이 예측 가능성이 디버깅을 쉽게 만들어 줍니다.
 
#### 7. Redux의 전체 흐름
 
이 세 원칙이 합쳐지면 Redux의 흐름이 완성됩니다.
 
```
사용자가 버튼 클릭
  -> Action 발행: { type: 'PLUS' }
  -> Reducer가 (현재 상태, Action)을 받아서 새 상태를 리턴
  -> Store에 새 상태 저장
  -> 상태를 구독하고 있는 컴포넌트가 리렌더링
```
 
이 흐름에서 상태가 바뀌는 경로는 항상 하나입니다. **Action -> Reducer -> Store** 예외가 없습니다.  
어디서 상태가 바뀌었는지 추적하려면 Action 로그만 보면 됩니다.  
Action은 단순한 객체이기 때문에 저장할 수 있고, 저장할 수 있으면 "되감기"가 가능합니다.
 
그렇다면 어떻게 되감기가 가능할까요?
 
Reducer가 순수 함수이고, 기존 상태를 수정하지 않고 새 객체를 만들어 리턴하기 때문입니다.  
모든 상태가 별도의 객체로 남아 있어서, 상태의 전체 역사가 보존됩니다.
 
```jsx
import { createStore } from 'redux';
 
function counterReducer(state = { count: 0 }, action) {
  switch (action.type) {
    case 'PLUS':
      return { count: state.count + 1 }; // 기존 state를 수정하지 않고, 새 객체를 만들어 리턴
    case 'MINUS':
      return { count: state.count - 1 };
    default:
      return state;
  }
}
 
const store = createStore(counterReducer);
 
// 상태의 역사가 쌓이는 과정
store.dispatch({ type: 'PLUS' });  // 상태 1: { count: 0 } -> 상태 2: { count: 1 }
store.dispatch({ type: 'PLUS' });  // 상태 2: { count: 1 } -> 상태 3: { count: 2 }
store.dispatch({ type: 'MINUS' });  // 상태 3: { count: 2 } -> 상태 4: { count: 1 }
```
 
매번 새 객체를 만들어 리턴하기 때문에, 이전 상태가 사라지지 않습니다.  
Redux DevTools는 이 점을 이용하여 상태의 전체 역사를 기록합니다.
 
```jsx
// Redux DevTools의 핵심 원리
// (실제 코드는 미들웨어/enhancer로 감싸져 있지만, 원리는 동일합니다)
// 위의 코드 counterReducer로 만든 store를 기준으로 설명합니다.

const history = [];
let currentIndex = -1;

// Store가 생성될 때 초기 상태를 기록
history.push(store.getState());
currentIndex = 0;

// store.dispatch()가 호출될 때마다 상태를 기록
const originalDispatch = store.dispatch;
store.dispatch = (action) => {
  originalDispatch(action);           // 원래 dispatch 실행
  history.push(store.getState());     // 새 상태를 history에 push
  currentIndex++;
};

// 사용 예시
store.dispatch({ type: 'PLUS' });  // history[1] = { count: 1 }
store.dispatch({ type: 'PLUS' });  // history[2] = { count: 2 }
store.dispatch({ type: 'MINUS' });  // history[3] = { count: 1 }

// time travel — 원하는 시점의 상태를 꺼냄
// DevTools는 history[index]를 꺼내서 UI에 반영
const pastState = history[2];  // { count: 2 } — 이 시점으로 되감기
currentIndex = 2;
```
 
만약 기존 상태를 직접 수정하는 방식이었다면 어떨까요?
 
```jsx
// Flux — 변수를 직접 수정
let count = 0;

AppDispatcher.register((action) => {
  if (action.type === 'PLUS') {
    count = count + 1;  // 기존 변수를 덮어씀
    // count가 0이었는데 1이 됨. 0은 사라짐.
  }
  if (action.type === 'MINUS') {
    count = count - 1;  // 기존 변수를 덮어씀
    // count가 1이었는데 0이 됨. 1은 사라짐.
  }
});

// history에 넣고 싶어도 이전 값이 덮어써져서 없음
// time travel 불가능
```
 
기존 방식은 값을 덮어쓰기 때문에 이전 상태가 사라집니다. 되돌릴 수 없습니다.  
Redux는 불변성<u style="color: red; font-weight: bold;">(기존 상태를 수정하지 않고 새 객체를 만들어 저장)</u>을 강제하여, 이전 상태를 보존하고 되돌리기를 가능하게 만들었습니다.  
이것이 Redux DevTools에서 "time travel debugging"이 가능한 이유입니다.
 
#### 8. 정리하며...
 
"상태를 어떻게 예측 가능하게 관리할 것인가?"
 
이 질문에 대한 답은 결국 웹 개발의 복잡성이 커지면서 자연스럽게 만들어졌습니다.  
MVC 패턴에서 상태가 여기저기 튀던 문제를 Flux가 "단방향 흐름"으로 잡았고, Redux가 그것을 "하나의 Store, 순수 함수, 불변성"이라는 원칙으로 더 단순하게 정리한 것입니다.

이는 이전 글 중 var/let/const를 다루며 "잘못 만든 게 아니라 당시에는 합리적이었다"고 썼던 것처럼, MVC도 잘못된 패턴이 아니었습니다. 단순한 앱에서는 충분히 잘 동작했습니다.

공통점은 "당시에는 합리적이었는데, 규모가 커지면서 한계가 드러났고, 그 한계를 해결하기 위해 새로운 구조가 탄생한 것"입니다.

다만 Redux도 완벽한 해답은 아닙니다. 보일러플레이트가 많다는 비판, 작은 앱에서는 과하다는 지적이 있었고,
