개발 공부를 하면서 매일 기록을 남기고 있으신가요?   
기록 서비스는 많지만, 꾸준히 의미있는 학습글을 가져가는 것은 어렵게 느껴집니다.

이번 글에서는 이 문제를 해결하기 위한 서비스, "DevLog"에 대해 기획 해 보고자 합니다.

---

#### 1. 왜 DevLog를 만들려고 할까요?

##### 매일 학습 기록(잔디 깔기)을 진행하면서, 발견한 문제점

> 얼마전 잔디깔기를 하면서 Next.js에 대해 열심히 공부하고 기록을 남겼습니다.  
> 그런데 멘토께서 "Next.js의 이 기능"을 왜 공부하셨나요? 라고 물어보시니 대답이 막혔습니다.
>
> 분명히 기록은 했는데, '**학습목적**'이 명확하지 않았던 것입니다.  
> 남들이 좋다고 쓰길래 썼는데, 정작 학습목적에 대해서는 생각해보지 않았던 것이 컸습니다.
>
> 또한 학습목적이 명확하지 않다보니, '**기억**'이 나지 않았고, 자연스레 글 쓰는것을 멈추게 되었습니다.   
> 그렇다고 매번 학습목적까지 생각하자니 '**작성시간**'이 오래 걸렸습니다.
>
> Git, Tistory, Velog 등등.. 작성은 편했지만, 이 문제를 해결해주진 않았습니다.

##### 무엇을 생각해 봐야 할까요?
- UX의 대상이 되는 <u style="color: red; font-weight: bold;">주 타겟을 정의</u>하고 <u style="color: red; font-weight: bold;">위에서 발견한 문제를 구체화</u> 해야 합니다.
- 그래야 구체적으로 생각한 부분을 통해, 솔루션을 제시할 수 있습니다.

---

#### 2. Who - 누가 쓸까요?

##### **주 타겟**
- 자기개발을 위해 학습기록을 남기려는 개발자

##### **문제 구체화 : 기록이 어려운 이유**
- **학습 목적** - 본인이 <u style="color: red; font-weight: bold;">왜 이걸 공부하는지 명확하지 않으면</u>, 학습 방향이 흐려지고 동기가 약해집니다.
- **학습 기억** - 또한 시간이 지나면 <u style="color: red; font-weight: bold;">뭘 공부했는지, 왜 공부했는지 기억이 희미해집니다.</u>
- **학습 지속성** - 목적과 기억이 없으면 <u style="color: red; font-weight: bold;">"오늘도 써야 하나?" 회의감이 들어 기록이 끊깁니다.</u>
- **작성 시간** - 매번 학습목적까지 고민하며 쓰려면 <u style="color: red; font-weight: bold;">기록 한 번에 시간이 오래 걸립니다.</u>

##### **솔루션**

DevLog에서 '작성 인프라'를 제공 -> '학습목적', '기억', '지속성', '작성시간' 문제를 완화해줍니다.

Solution 1. **목표 설정 기능** 제공  
   -> <u style="color: red; font-weight: bold;">'학습목적'</u>을 명시적으로 적어두고, "매일 상기"  
   -> <u style="color: red; font-weight: bold;">'목표별로 글을 분리'</u>하여 "학습내용"이 뒤엉키지 않도록 함

Solution 2. **최근 기록 타임라인** 제공  
   -> <u style="color: red; font-weight: bold;">'기억'</u> — 학습목적 상기와 함께 "어제 뭘 공부했더라?"를 바로 확인가능

Solution 3. **연속 기록 통계** 제공  
   -> <u style="color: red; font-weight: bold;">'지속성'</u> — 목적과 기억이 담긴 글 + 연속기록 일수를 보고 "오늘도 써야지"라는 동기가 생김

Solution 4. **기록유형(TIL, DeepDive, TechStudy) 양식** 제공  
   -> <u style="color: red; font-weight: bold;">'작성시간'</u> — 학습 목적을 대신 고민해줄수는 없지만, 양식을 통해 "뭘 쓸지를 선행 제시"하여 시간절약

---

#### 3. What - 무엇을 만드는가?

##### 1단계 : 기록유형(TIL, DeepDive, TechStudy)에 맞게 양식을 제공
'작성시간(Solution 4.)' 문제를 해결하기 위해,  
기록 유형별로 양식을 미리 제공하여 "무엇을 적을지"를 선행 제시 

![DevLog 양식](../../../assets/20260406/form.png)

**TIL (Today I Learned)** 
- 목적 : 오늘 배운 것을 가볍게 기록
- 양식 : 짧은 텍스트 + 태그(#React, #비동기 등) + 날짜

**DeepDive** 
- 목적 : 특정 기술을 깊이 파고든 학습 기록
- 양식 : 주제 + 학습 동기 + 핵심 내용 + 알게 된 점

**TechStudy** 
- 목적 : 강의/책/아티클 등 학습 자료 기반 기록
- 양식 : 자료 출처 + 요약 + 본인 생각/의문점

**공통 기능**
- 기록 수정/삭제
- 기본 검색 (태그/키워드 기반)

**사용 예시**  
- 1일차, 유지/보수 중 'React Hook 개념에 대해 공부할 필요성'을 느낌 -> TIL에 간략히 기록 
- 2일차, 어제 TIL을 보고 본격적으로 공부하려고 했으나, 업무에 쫓겨 간략히 내용만 확인 -> TIL에 간략히 기록
- 3일차, React Hook에 대한 강의수강 -> TechStudy에 기록
- 4일차, TechStudy를 작성하다보니, React코어 부분까지 궁금하게 됐음 -> DeepDive에 기록
- ?일차, 유저 편의에 맞게 양식을 제시

##### 2단계 : 대시보드 & 위젯

**대시보드** 
- 상단에 위젯 4개가 고정 배치되고, 그 아래로 작성한 글 목록이 이어집니다.

![DevLog 레이아웃](../../../assets/20260406/layout.png)

**위젯** 
- **연속기록 위젯** : 연속 기록일, 최고기록일을 표시  
  -> '지속성(Solution 3.)' : 끊기면 아쉬운 느낌을 줘서 기록 지속을 유도

- **목표 리스트** : 현재목표 이외의 목표를 목록으로 표시  
  -> '목적별 글분리(Solution 1.)' : 목적에 따른 글을 구분하여, 학습내용이 뒤엉키지 않도록 함

- **학습목표 위젯** : 현재 목표와 이유를 표시,   
  -> '학습목적(Solution 1.)' : 왜 공부해야 하는지? 이유를 명시하여, 학습목적을 상기시킴

- **최근 기록 타임라인** : 최근 기록을 시간순으로 요약해서 보여줌  
  -> '기억(Solution 2.)' : "어제 뭘 공부했더라?"를 대시보드에서 바로 확인

#### 4. What Not — 무엇을 안 만드는가?

| 기능 | 제외 이유 |
|------|-----------|
| **랭킹** | 꾸준한 글 작성유도는 가능하나, 개인학습 목적이기에 필수는 아님 |
| **외부 블로그 연동** | 외부 서비스(Git, Tistory, Velog)에 옮길수 있게 txt, mdx등으로 컨버트 하는 기능. 필수는 아님|
| **AI 글 다듬기** | 편의성은 좋으나, LLM모델 및 비용을 생각해야 함 |
| **AI 글 검색** | 자연어문장을 통한 검색을 통해, 편하게 찾을수 있지만, DB기반 검색으로도 충분함 |

#### 5. How — 기술 스택
 
##### 언어, 라이브러리 : TypeScript + React
 
- TypeScript의 타입 시스템으로 API 응답, 위젯 상태, 통계 계산 로직의 구조를 미리 명시해 두고,  
약속과 다른 값을 쓰면 IDE/빌드 단계에서 바로 오류를 잡아 리팩토링 내성을 높이고 런타임 버그를 줄이고자 한다.

- React는 SPA · 대시보드 · 폼 등 상태 기반 UI를 컴포넌트 단위로 구성하는 데 적합한 라이브러리이다.

- 그러나 React가 유일한 선택지는 아니며, Svelte나 SolidJS처럼 더 작은 번들 크기와 더 빠른 DOM 업데이트를 제공하는 대안도 존재한다.

- DevLog는 위젯 4개와 간단한 클라이언트 상태(목표 선택, 모달, 필터) 정도의 규모로, 경량화 번들로 봤을때는 대안이 더 어울릴 수 있다.

- 다만, 프론트엔드의 실무 생태계를 고려했을 때, 비중이 높은 React의 숙련도를 어필 할 필요가 있어 React를 선택하게 되었다.
 
##### 프레임워크: Next.js

- DevLog는 로그인 기반이라 SEO 이점은 없다.

- 그러나 Next.js의 SSR + CSR 조합을 활용하면:
  -> 첫 화면은 SSR로 빠르게 보여주고 (TTV 단축)
  -> 이후 페이지 이동은 CSR로 블링크 없이 부드럽게

- 다만 SSR의 TTV & TTI 공백은 Skeleton UI(로딩 중 회색박스를 보여줌)로 체감 개선 예정

- 추가로 API Routes, 파일 기반 라우팅 등 생산성 도구도 함께 사용할 수 있다.
