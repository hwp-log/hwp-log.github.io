## 1. Hook은 항상 컴포넌트 최상단에서, 매 랜더링마다 같은 순서로 호출한다. {#hook-component-render}

[공식문서](https://legacy.reactjs.org/docs/hooks-rules.html)의 내용에 따르면, "반복문, 조건문 또는 중첩함수"내에서 Hooks를 호출하지 말라고 한다. 이런 이유는 무엇일까? 아래의 예시를 통해 이해한 내용을 설명하고자 한다.

```jsx
    function Form() {
        // 1. 상태저장 Hook 호출(name state)
        const [name, setName] = useState('Mary');
        // 2. 후처리 Hook 호출
        useEffect(function persistForm() {
            localStorage.setItem('formData', name);
        });

        // 3. 상태저장 Hook 호출(surname state)
        const [surname, setSurname] = useState('Poppins');
        // 4. 후처리 Hook 호출
        useEffect(function updateTitle() {
            document.title = name + '' + surname;
        });
    }
```

상기의 예시를 보면, 1~4까지 순서대로 기재되어 있음을 확인 할 수 있다. 물론 내용을 보면 각 상태를 저장하는 변수 name, surname으로 나뉘어져 있어 [1,2],[3,4]로 구분해서 생각할 수 있다. 그러나 지금은 1,2,3,4 순서에 집중해 보도록하자.

```jsx
// 순서 1
useState('Mary')
// 순서 2 
useEffect(persistForm)
// 순서 3
useState('Poppins')
// 순서 4
useEffect(updateTitle)
```

해당함수들은 ['랜더링이 일어날 때마다 컴포넌트 함수 전체가 통째로 재실행'](#rerender-entire-function)하게 된다. 여기서 주의해야 할 점은, 정해진 순서가 누락될 수 있다는 위험요소가 있다는 것이다. 그렇다면 순서가 누락될 위험요소는 무엇일까?

```jsx
    function Form() {
        // 1. 상태저장 Hook 호출(name state)
        const [name, setName] = useState('Mary');
        // -> 갱신(조건문 추가)
        // 2. 후처리 Hook 등록
        if (name !== '') {
            useEffect(function persistForm() {
                localStorage.setItem('formData', name);
            });
        }
        // 3. 상태저장 Hook 호출(surname state)
        const [surname, setSurname] = useState('Poppins');
        // 4. 후처리 Hook 등록
        useEffect(function updateTitle() {
            document.title = name + '' + surname;
        });
    }
```

갱신된 코드 내용에 주목하자. 조건문이 useEffect훅을 감싸고 있다. 해당 내용이 실행된다면 어떤일이 발생할까? 조건에 name 내용이 존재하지 않는다면, 어떻게 될까?

```jsx
// 순서 1
useState('Mary')

// 순서 2?? 
//useEffect(persistForm) 작동 건너뜀!!

// 순서 2 (이전 순서 3)
useState('Poppins') // 순서3번 이였던 hook이 2번으로 작동해서 오작동!!
// 순서 3 (이전 순서 4)
useEffect(updateTitle) // 순서4번 이였던 hook이 4번으로 작동해서 오작동!!
```

재밌게도 이전순서가 작동을 건너뛰자, 후순위 작동이였던 훅들이 앞 당겨졌다. 즉, '하나의 컴포넌트에서 랜더링이 일어날때마다 전체가 다시 작동한다는 원칙'이 전제로 깔려있었기 때문에, 인덱싱 된 순서가 지켜져야 되는데 그렇지 못해서 오작동이 발생 한 것이다.

보통은 상태가 저장된 변수의 갱신을 통해 해당 내용만 작동한다고 생각하지만, 사실은 컴포넌트 안의 상태변수가 어떤것으로 변하던, 전체가 갱신되는 일이 발생해서 생기는 것이다. 'React는 지정해 놓은 전체 Hooks를 순서대로 실행시킨다는 것을 기본전제라고 생각하자.' 그러면 훅 호출 자체를 조건문으로 감싸는 방식이 모순이 되니 사용할 일이 없어질 것이다.

그렇다면, 이 경우는 어떠할까?

```jsx
    // ...
    // 2. 후처리 Hook 호출
        useEffect(function persistForm() {
            if (name !== '') {
                localStorage.setItem('formData', name);
            }
        });
    // ...
```

Hook은 항상 실행시키되, name의 내용이 있다면, 후처리 하겠다는 것이다. 이것이 훅의 전체실행이라는 최소조건을 총족하되... 조건문, 반복문, 중첩함수를 사용할 수 있게 해주는 핵심요소 인 것이다.
이렇게 조건을 총족하게 된다면 React 공식문서에서 말한 'Hook은 항상 컴포넌트 최상단에서, 매 랜더링마다 같은 순서로 호출' 조건을 만족시킬 수 있다.

## 2. React는 랜더링이 다시 일어날 때마다 컴포넌트 함수가 통째로 재실행. {#rerender-entire-function}

그렇다면 랜더링이 다시 일어날 때마다 컴포넌트 함수 전체가 재실행 되는 것일까? 이는 React의 주요 역할을 생각해보면 알 수 있다. React는 어플리케이션 상태에 따라 UI자체가 항상 최신 상태로 유지 하고자 한다. 정확히 말하자면, 상태가 변화했을때 화면 전체가 최신상태로 변하고, 어떤 곳이 변경되었는지 파악하여 계산한뒤, 갱신하는 것이다.

이 과정은 크게 3단계로 나눌 수 있는데

1. 트리거링 : 상태가 변경될 때 발생.
2. 랜더링 : React가 컴포넌트의 구성 Hook을 전체 작동시키고 새로운 UI를 다시 계산.
3. 커밋 : 이렇게 계산된 UI와 이전 상태를 비교해, 변경된 DOM만 바꿔준다.

여기서 주의할 점은, 1의 useState(...)의 인자가 들어가서 초기값이 변할때 인데, 여기서 인자로 들어가서 바로 훅이 실행되는 것이 아니라, 인자 -> 큐 예약상태 -> 컴포넌트 리랜더 -> 훅 실행 순으로 작동한다는 것이다.

이렇듯 useState에서 상태값이 들어간다고 해서 바로 훅의 상태에 적용되는것이 아닌, '큐 -> Re-render과정'을 거쳐서 작동한다는 것에 주목해야 한다. 헷갈리겠지만 아래의 코드를 보면 이해가 쉬워진다.

```jsx
    // 0. setName(...) 또는 setSurname(...) 호출 (업데이트 발생)
    // 1. 해당 state 훅의 업데이트가 큐에 예약됨
    // 2. Form 컴포넌트 리렌더 시작
    function Form() {
    // 3. 예약된 상태저장 Hook 호출 (name state 읽기)
    const [name, setName] = useState('Mary');
    // 4. 후처리 Hook 등록 (이번 렌더 후 실행할 effect)
    useEffect(function persistForm() {
        localStorage.setItem('formData', name);
    });
    // 5. 상태저장 Hook 호출 (surname state 읽기)
    const [surname, setSurname] = useState('Poppins');
    // 6. 후처리 Hook 등록
    useEffect(function updateTitle() {
        document.title = name + '' + surname;
    });
    }
    // 7. DOM 커밋 후, 4,6에서 등록했던 useEffect 콜백들이 실행
```

이것이 "React는 랜더링이 다시 일어날 때마다 컴포넌트 함수가 통째로 재실행" 되는 실제 순서이다.

## 3. useEffect의 실행시점, 의존성 배열의 이해

그렇다면 상태저장 Hook인 useState의 작동원리를 알았다면, 후처리 Hook인 useEffect의 작동원리는 무엇일까? useEffect는 작동시점을 기준으로 설명이 가능한데, 이는 2가지로 나누어서 얘기해 볼 수 있다.

1. DOM이 준비되었을 때(초기 랜더링)
2. 의존성 배열의 변화가 있을 때

먼저, "1. DOM이 준비되었을 때"를 설명 해 보겠다.

```jsx
    import React, {useEffect} from "react";
    // ...
    export default function App() {

        useEffect( () => {
        // 2.컴포넌트가 처음 렌더링되고, 그 결과가 DOM에 반영된 "뒤에" 한 번 실행되는 콜백
        alert('DOM이 완전히 로드 되었습니다.')
        },[])

        return (
            // 1.컴포넌트의 UI가 DOM에 렌더링되는 단계
            <div>안녕하세요! user님!</div>
        );
    }
```

코드에서 주석에 달린 순서대로 작동하는 것에 주목해보자. 이는 useEffect함수의 끝쪽에 달려있는 '의존성 배열 []'가 작동하지 않았을 때를 말한다. 즉, 초기에는 의존성 배열의 변화없이 초기에 App컴포넌트가 작동했을때의 작동을 설명한 것이다. 이 경우의 Dom은 return되는 <div>를 말하며, <div>가 DOM에 반영된 이후, 콜백으로서 alert의 내용이 실행된다. 

그렇다면 "2.의존성 배열의 변화가 있을 때"는 어느 경우를 말하는 것일까?

```jsx
    import React, {useState, useEffect} from "react";
    // 1. 해당 state 훅의 업데이트가 큐에 예약됨
    // 2. App 컴포넌트 리렌더 시작
    export default function App() {
        // 3. 리랜더링 과정에서 useState 훅 호출 (name state 읽기)
        const [value, setValue] = useState('user');    

        useEffect( () => {
        // 5. DOM 커밋 이후, 의존성 배열의 이전 값과 현재 값이 다르면 실행되는 콜백
        alert('DOM이 완전히 로드 되었습니다.')
        },[value])

        return (
                // 4.컴포넌트의 UI가 DOM에 렌더링되는 단계
                <div>안녕하세요! {value}님!</div>
                // 0. setValue(...) 호출 (업데이트 발생)
                <button onClick={() => setValue("user1")}>user1로 변경</button>
        );
    }
```

"1번 DOM이 준비되었을때"랑 다르게 [value]라는 내용이 바뀌게 된 것을 알 수 있다. 즉, setValue를 통해서 value값이 바뀌고 그 value값이 useEffect의 의존성 배열로 작동하면서 상태가 바뀌게 되는 것이다. 여기서 의존성 배열은 담고 있는 상태를 말하는 것인데, 한개가 아닌 여러개가 들어갈 수 있다. 그 여러개가 들어간 배열의 내용 중 하나가 바뀐다면, DOM의 변경 내용과 상관없이 alert문이 실행된다는 것이다.

## 결론

결론적으로 Function함수의 최소조건

1. Hook은 항상 컴포넌트 최상단에서, 매 랜더링마다 같은 순서로 호출.  
2. 랜더링이 다시 일어날 때마다 컴포넌트 함수가 통째로 재실행.  
3. useEffect의 실행시점, 의존성 배열의 이해.  

이 세 가지 최소조건이 생긴 근본적인 이유는 2번, 즉 컴포넌트 함수가 렌더링이 일어날 때마다 통째로 재실행되는 구조 때문이다. 통째로 재실행되기 때문에 React는 Hook을 호출 순서로 식별할 수밖에 없고, 그래서 1번의 순서 보장 규칙이 필연적으로 따라온다. 또한 통째로 재실행된 결과가 DOM에 커밋된 이후에야 후처리가 가능하기 때문에, 3번의 useEffect는 DOM 커밋 이후라는 실행 시점과 의존성 배열 비교라는 실행 기준이 분리되어 작동하는 것이다.
